[edit] Feature Overview

IPA update mechanism provides one update tool, /usr/sbin/ipa-lapd-updater to help customer upgrade their systems, including IPA server and replica servers, to a higher version. After the whole system finished upgrading, user and group date will remains the same as before. The schema might be updated, but it will compatible with previous schema.

IPA server upgrading is a multi-step processing. And there is no roll back once IPA server upgrade from lower version to higher version. This test plan contains test cases to ensure IPA update will perform well in all platforms and platform combinations.

This update mechanism is a long term solution for IPA upgrading. A mini upgrading language is defined in this update mechanism. The update script will use update configuration file (/usr/share/ipa/updates), follows the rules in this mini language to upgrade IPA server. A test for this mini language is also included in this test plan

[edit] Test approach

[edit] General approach for each test cases

In a nutshell, to verity the success of upgrading, the following logic applies to each test cases

1. Set up test host
2. Perform upgrading according to the recommended procedure as below: 

1. yum update ipa-server
2. /usr/sbin/ipa-ldap-updater -t
3. echo $?
4. if value == 2
   /usr/sbin/ipa-ldap-updater

3. Verify the success of upgrading

To verity the success of upgrading, one good solution is take a "snapshot" of database (using ldap2db) right before and after server update. And we can compare these two snapshot txt file to verity whether the update changed something it should NOT.
[edit] Set up Host

In the real world, customer can have many different ways to setup their IPA server and client hosts. But QE can only certify these that follows our documentations. The term set up test host refers to the following procedure:

  • If this is IPA server setup
    1. Using a clean machine that has fresh RHEL 5 i386 or RHEL 5 x86_64 installed
    2. Subscript machine to RHN plus IPA channel
    3. yum install IPA server
    4. run ipa-server-setup to setup IPA server
  • If this is IPA client setup
    • For the following OS platforms
      • RHEL 5 i386 & x86_64
      • RHEL 4 i386 & x86_64
      • follow the next 4 steps
        1. Using a clean machine that has fresh OS installed
        2. Subscript machine to RHN plus IPA channel
        3. yum install IPA client
        4. run ipa-client-setup to setup IPA client
    • For the following OS platforms, please following (RedHat) official IPA client installation guide
      • RHEL 3 i386
        • (What happens to RHEL 3 x86_64?)
      • RHEL 2.1 i386
        • (What happens to RHEL 2.1 x86_64? Does it exist, need find it out)

[edit] Client Certify test

The purpose of Client Certify test set is to certify the basic function of IPA client host. It contains the following test cases:

  1. Client configuration test
    • this set of test only applies to the following platform
      1. RHEL 5 i386
      2. RHEL 5 x86_64
      3. RHEL 4 i386
      4. RHEL 4 x86_64
  2. Client host authentication test
    • this set of test only applies to the all supported platform
    • run kinit <IPA user> to get authenticated
      • verify the authentication by running ipa-moduser to modify user's email address
[edit] Server Certify test

It is necessary to ensure IPA server behaves correctly right before test starts. This is a small set of test to ensure the test environment. It contains the following test cases (and they all have to pass)

  1. start and stop
  2. as admin, create a new user account
  3. as user, modify its email address
  4. as user, change its password
  5. as admin, in-active a user (lock a user)
  6. as admin, active a locked user (unlock a user)
  7. as admin, modify a user's home directory
  8. as admin, reset user's password
  9. as admin, create a new group account
  10. as admin, add user to group
  11. as admin, remove user from group
  12. as admin, delete a user account
  13. as admin, delete a group
[edit] Server Identical test

The challenge of upgrade test is that we have to find a way to certify the upgraded system identical to they previous system. In another words, we have to define a set of test to prove the success of upgrading. To achieve this goal, a set of test grouped here:

  1. all user data. And we should pay special attention to the following:
    • user's authentication information (including password, certificate)
    • user's rights information (such as editor information)
  2. all group data
  3. DNA information (not sure about this one)
  4. all relations, such as memberOf
  5. schemas
    1. if not modified, should same as previous server
    2. if modified (this should be documented), should compatible with previous server
  6. all server configuration, such as password policy (there must be more configuration related details, please give me some ideas) should has no change
[edit] Data Set used in this test plan

There are total 3 data set used in this test plan. The detail definitions are below:

  • small data set
    • total users: 100
    • total groups: 10
    • distribution:
      • there are total 5 nested group layers. Each layer will has exactly 2 groups
      • each group will contains 10 users
      • no cycling group setup
      • each group is only direct member of group(s) that exactly one layer up. such as if group g-1-1 has direct member g-2-1 and g-2-1 has a direct member g-3-1, then, group g-3-1 will not setup as direct member of group g-1-1
  • middle data set (simulating real world situation)
    • total users: 150,000
    • total groups: 150
    • distribution:
      • there are total 5 nested group layers. Each layer will has exactly 30 groups
      • each group will contains 1,000 users
      • no cycling group setup
      • each group is only direct member of group(s) that exactly one layer up. such as if group g-1-1 has direct member g-2-1 and g-2-1 has a direct member g-3-1, then, group g-3-1 will not setup as direct member of group g-1-1
  • large data set (used in extreme test case)
    • total users: 2,000,000 (2 million users)
    • total groups: 1,000
    • distribution:
      • there are total 100 nested group layers. Each layer will has exactly 10 groups
      • each group will contains 2,000 users
      • random group-user relation setup

[edit] In the Real World (Negative Test Cases)

In a real world, we have to consider the following major negative situation:

  1. user interruption
  2. physical machine power failure
  3. network difficulty (Network outage)

Even IPA upgrading is a one way upgrading, we still have to consider what we should do during above situation to maintain a good user experiences. I have seen no section in design doc talk about this. I will make up some scenarios here, and hopefully we have answers very soon.
