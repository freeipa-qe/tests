[edit] Feature Overview

There are total two NIS relate plug-ins. The first on is NIS server plugin. It functions as a replacement of traditional NIS server. Another one is "schema compatibility plug-in". The purpose of this plug in is to translate some schema conflict between Solaris IPA client and IPA server. The test for this plug in is in Additional test section.

As of IPA 1.1 release, there is NO NEED to test the NIS server plugin. Only test cases in Additional test would be run. (The NIS server plug-in code has been finished. The code will ship this time, but QE will not certify this part of code)


[edit] IPA NIS plugin Overview

IPA NIS plugin is a Red Hat Directory Server plugin that behaves as a fully functional NIS server. It should response to NIS client no different from a traditional NIS server, except the functions which are only used by the replica part in a replication setup (YPPROC_CLEAR, and YPPROC_XFR). The plugin implementation is pretty much a Red Hat re-writing of NIS server.

From NIS client's perspective, IPA NIS plugin should meet the "common understanding" of what a traditional NIS server should do. In another words, customer should configure their NIS client as what they do usually, run NIS commands as usual, and they should notice no difference at all. A set of test that using NIS commands is a basic test set.

From IPA server's perspective, this plugin should server all IPA users as well. A set of testing around how this plugin response to IPA client request is also a must.

[edit] Traditional NIS server

In a nutshell, the Network Information Service (NIS) provides a simple and centralized network looking up service to easy network accounts and groups management. Its purpose is to provide information, that has to be known throughout the network, to all machines on the network. Information likely to be distributed by NIS is:

  • login names/passwords/home directories (/etc/passwd)
  • group information (/etc/group)
  • host names and IP numbers (/etc/hosts)

Once NIS server has user and group information, all NIS client machine will be able to communicate with NIS server to assign proper network-wide resource to its users and groups.

IPA NIS plugin should behave exactly as above.

[edit] Test approach

Two sets of test will be executed:

The first set of test is functional testing. The purpose of this set is to verity IPA NIS plugin behave.

By using NIS commands to configure NIS client and verity the NIS server response, we can prove whether IPA NIS plugin behave same as a real NIS server. This set of testing can be break into the next 2 categories:
  1. NIS client configuration.
  2. NIS client command execution testing.

The second set of test is IPA specific functionality testing. The purpose of this set of test is to verity the plugin implemented as design and IPA specific topics. It including the following areas: (please check out test cases part for more details)

  1. IPA specific functional testing
  2. Security testing
  3. IPV6 testing
  4. Stress testing
  5. Performance testing

[edit] Additional test: schema compatibility plugin test

[edit] What is this

This plugin is designed specifically for IPA client on Solaris platform to enable NIS function with IPA server.

A special plug in is required because on Solaris platform, NIS client is looking for "member" for user-group relation. And since we already use this attribute in IPA schema to map between IPA user and groups, we have to make a special mapping for Solaris NIS client.

[edit] Environment Setup

  • Server side configuration:
    • The schema compatibility plugin is part of server and it is automatic configured during server installation time. There is no special server side configuration need to be done
  • Client side configuration:
    • This is a plug in that designed specifically for Solaris platform. It works for Solaris 8, 9, 10 (SPARC) and Solaris 10 (x86 arch). The configuration guide here only apply to these 4 platforms
    • Steps to configure (assuming we already have IPA server installed)
      1. edit /etc/nsswitch.ldap and under the hosts and ipnodes entries make them look like:
        • hosts: files dns
        • ipnodes: files dns
      2. run: ldapclient init ipa.example.com (IPA server)
      3. ps -ef | grep nscd and kill it
      4. getent passwd admin

[edit] Risk analysis (what won't be tested)

[edit] This is acceptance test set in this release cycle

All test cases in this test plan are just Acceptance level test cases. A fully coverage of NIS functional testing is a required. However, due to the time and resource we have right now, we won't be able to implement a complete and comprehensive testing set. As of this release (IPA 1.1), QE will verity the basic NIS client and server functionality. More corner and negative test cases will be added as we have more time at IPA 2.0 release cycle.

[edit] NIS Master-slave setup/configuration won't be tested

Traditional NIS server can setup Master-Slave relation. And this feature is not implemented in current IPA NIS plugin release. To be specific, the following 3 theoretical-possible deployment will NOT be tested.

  1. UNIX NIS Master <==> IPA NIS Plugin Slave
  2. IPA NIS Plugin Master <==> UNIX NIS Slave
  3. IPA NIS Plugin Master <==> IPA NIS Plugin Slave

= The Schema Compatibility Plugin in IPA =

RFC 2307 defines object classes and attributes suitable for representing
various POSIX-related information in a directory.  The "posixGroup"
object class, for example, models a group of accounts, including the
POSIX group's ID (as the value of the "gidNumber" attribute) and the
names of each of the members of the group (as values of the "memberUid"
attribute).

This representation is supported by the native LDAP client software
supplied with a number of Unix and Unix-like operating systems.

The draft revision of RFC 2307, known as "rfc2307bis", modifies the
contents of a group so that it no longer contains the names of its
members in the "memberUid" attribute, but rather uses the "member"
attribute to list the distinguished names of the entries which represent
the individual members.  A client is typically expected to search the
named entries to determine the names of the group's member users.

It is this representation of groups which FreeIPA and RHEIPA use.

This representation is not as well-supported by the native LDAP client
software supplied with Unix and Unix-like operating systems.

When configured to consult a server which stores information this way, a
client which does not support reading group membership from the "member"
attribute will still able to look up a group and read its name and ID,
but will be unable to "see" the members of the group.

For these clients, an IPA server can be configured to use the Schema
Compatibility Plugin to resolve the group membership at the server.  The
plugin will examine the group entries stored in the server and, as
configured, read the distinguished names of the members of those groups.

It will examine the named entries and read from them the names of the
members of the group.  It will then use that data to create an in-memory
entry, elsewhere in the directory tree, which contains the group
membership information in a form which the client is able to process.

As configured, an IPA server provides this information, for groups whose
entries are beneath "cn=groups, cn=accounts, $SUFFIX", in an area
beneath "cn=groups, cn=compat, $SUFFIX".

  • Question: Do we need support the following? why and why not? Click the source URL above to find out the following terminology
    • DNS Forwarding Mode
    • Shadow Password Mode
    • Alternate Directory for the passwd File
    • Long Username and Long Hostname support
    • Support for NIS ypbind v3 Protocol

    • Alternate Directory for the passwd File

  • Answer 
DNS forwarding mode: no.
Shadow Password Mode: yes, but you have to reconfigure the entry formats.
Alternate Directory for the passwd file: not applicable.
Long Username and Long Hostname support: yes (no assumed limits).
Support for NIS ypbind v3 protocol: no.
  Do we have any clients that can use the v3 protocol?
Files for Resolving Map Nicknames: these files live on the client, so
  it's not affected by the server.
Multi-Homed Node: no.
