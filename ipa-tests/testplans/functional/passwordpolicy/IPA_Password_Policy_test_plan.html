[edit] Feature Overview

Password policy provides regulations for IPA server to verify user's password while it is being created. The regulation is enforced by KDC server. The actual policy is stored in LDAP server under "cn=config". A sample is below: 
dn: cn=accounts,dc=test,dc=com
objectClass: top
objectClass: nsContainer
objectClass: krbPwdPolicy
cn: accounts
krbMinPwdLife: 3600
krbPwdMinDiffChars: 0
krbPwdMinLength: 8
krbPwdHistoryLength: 0
krbMaxPwdLife: 7776000
One complete IPA password policy contains the following attributes:
Attributes Sample Value Data Range
password expiration notification (days) 1 don't know
Min password lifetime (hours) -1 don't know
Max password lifetime (days) 24 don't know
Min. number of Character classes 0 don't know
Min.length of password 8 don't know
password history sizes 2 don't know

[edit] Testing Environment

An IPA server instance with default configuration on supported OS will provide a necessary testing environment.
Supported OS platforms are below
  1. RHEL 5.2 i386 (32bit)
  2. RHEL 5.2 x86_64 (64bit)
If test cases execute on local machine, default IPA server is required. If test cases execute on remote machine, the packages: IPA-Client and IPA-admintools are required.

Setup default testing environment

The text setup default environment refers to the following steps
  1. setup clean OS RHEL5.2 32bit/RHEL5.2 64bit
  2. subscribe to RHN standard channel
  3. setup IPA yum repository
    • official release: baseurl=http://porkchop2.devel.redhat.com/nightly/latest-RHEL5.2-IPA-1.0/1.0/$basearch/os/IPA/
  4. download IPA server bits and dependencies: yum install ipa-server
  5. setup IPA server: ipa-server-install, accept all default settings

[edit] Approach & Methodology [Test design]

[edit] Test assumptions & dependencies

There is no assumption and dependency for this test

[edit] Test approach

When user creates/changes its password, KDC server lookup the current password policy and apply the regulations for the password. Password policy only applies to password change, including create and modification, after the policy has been created or modified. A case study is below:
Assuming we have such password policy
Min. Password Lifetime (hours): 1
Max. Password Lifetime (days): 90
Min. Number of Character Classes: 0
Min. Length of Password: 8
Password History Size: 0
  • At day 1 user "USER_A" has a 9 letters password "userApass".
  • At day 2, the Minimum length of Password changes to 10.
    • Then User "USER_A"'s password is still valid. The modified password policy does not apply to previous password.
  • At day 3, user "USER_A" changes his password, the modified password policy will apply. The new password has to meet "min. password length >=10" requirement.


  • Test result verification:
    • IPA Server 1.0 & 1.1 use LDAP server to store password policy and it use KDC server to enforce it. However, this design might change as we move to version 2. To simplify the test result verification, the following mechanism apply:
      1. kinit as one user with new password to verify its ticket
      2. calling "ipa-pwpolicy --show"

ipa52[07/26/08 03:20]/tmp> ipa-pwpolicy --show
Password Policy
Min. Password Lifetime (hours): 1
Max. Password Lifetime (days): 90
Min. Number of Character Classes: 0
Min. Length of Password: 8
Password History Size: 0

[edit] Risk analysis (what won't be tested)

There are two ways to modify password policy. One is using command line tool ipa-pwpolicy. Another is modify through Web GUI. Since the command line tool and the web GUI are all send same XMP-RPC request to IPA back-end, there is no need to test this feature through web GUI.
And at same time, there are some test cases in web GUI test plan cover this area.


[edit] Test cases & Result

There are 3 set of test: smoke test, acceptance test and functional test.

  1. Smoke test
    • Smoke test proves the existence of password policy in IPA server. Smoke test will only have one comprehensive test case that covers all attributes of one complete password policy. The purpose of this test is to prove that IPA server contains a password policy as defined in "ipa-pwpolicy" command.
  2. Acceptance test
    • Acceptance test proves IPA server can create a valid password policy and enforce it. Acceptance will test each and every attribute that included in password policy. The purposes are below:
    1. IPA server will behave correctly within valid data range
    2. IPA server can do basic error handling
  3. Functional test
    • Functional test proves IPA server is functional well under all extreme cases. Functional test is an extension of acceptance test. It will focus on edge data around the valid data range. And it will contains more negative test cases to verify the server behavior under some extreme conditions.
  4. System integration test
    • System integration test proves password policy will apply whenever it suppose to.

[edit] System integration Test

Password policy be should enforced under the following conditions:

  1. when a new user is being created
    • Test steps:
    1. setup default environment
    2. create one new user with invalid password as admin [Verify this would fail]
    3. create one new user with valid password as admin [Verify this should success]
  2. when an existing user first login to system, system as user to change its default password
    • Test steps:
    1. setup default environment
    2. follow above test step to create a new user with valid password
    3. kinit as the new created user
    4. verify that system prompt for password change
    5. verify that invalid password will be reject
    6. verify that valid password will be accept
    7. verify that after KDC accepted this password, user can be authenticated with the new password
  3. when an existing user changes its password before it expires
    • Test steps:
    1. follow steps in test 1 to create a new user
    2. follow steps in test 2 to create a new password for this new user
    3. kinit as this new user
    4. change password (run ipa-passwd)
    5. verify that invalid password will be reject
    6. verify that valid password will be accept
    7. verify that after KDC accepted this password, user can be authenticated with the new password
  4. when an existing user changes its password after system detects its password expired
    • Test steps:
    1. follow steps in test 1 to create a new user
    2. follow steps in test 2 to create a new password for this new user
    3. change system to ensure user's new password expired
    4. kinit as this new user
    5. verity that system notify user the password has expired
    6. verity that when user hits CTRL-C to exit, next time the same prompt will show up
    7. verity that when user enter new password, only valid can be accept
  5. when admin reset user's password
    • Test steps:
    1. follow steps in test 1 to create a new user
    2. follow steps in test 2 to setup a valid password for this new user
    3. kinit as admin, reset user's password (the only way i can think of doing reset is through WebGUI)
    4. kinit as this new user
    5. verify that system prompts user password has been reset (? prompt for reset or expire )
    6. verify that only valid password can be accept by IPA
  6. when admin reset a bulk of user's password
    • can we even do this?


There are special rules when Directory Manager resets passwords. Please check with Rob. Dpal 08/11/08

[edit] userPassword attribute test

The current design doesn't prevent user access LDAP to change their data. Even we only expose the IPA utility for user to change their password, using ldap command to update user's data is legal as well. The test for userPassword is to ensure modification on this attribute will not cause any conflict or potential security risk.

  • [Req14] General UI Requirements

  • [Req19] Policy
    • [Req19.2] The UI should allow administrators the ability to set the Password policy, e.g. Password aging, quality (min/max/complexity rules for all passwords)
      • [Req19.2.1] This functionality should match that offered by RHDS8.0
      • [Req19.2.2] Password Quality should be enforced regards of what mechanism is used to change the password
  • [Req24]GUI Server Policies In general the GUI is not to entertain general management of the LDAP server, in fact a goal is to disguise the fact that an LDAP server exists underneath. However, some items as they pertain to the management of users and groups are necessarily server wide:
    • [Req24.1] Allow setting of system wide password policy
    • [Req24.2] We will NOT support fine grained password policy in this version)
