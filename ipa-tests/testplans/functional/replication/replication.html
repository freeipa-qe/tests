Feature Overvie

  1. server resource operation verification 
         * ssh
         * system login
         * nfs
         * failover
         * password operation:
               o self-change
               o administrator reset
               o expiration notice
  1. record (entry) operation verifycation
  2. membersof plugin verification
  3. enhanced replica test (that based on existing DS replica testing, but we don't execute them for now) 
Notes: all of above test need perform under 2MMR and 4MMR topology


[edit]

Testing Environment

  • machine to use:
  1. IPA Master Server: ipaserver
  2. IPA Replica server: ipareplica1
  • machine configuration:
    1. machine is rhn getistered machine (run "rhn_register --nox" from command line to register testing machine with RHN)
    2. build yum repo
      • rhel5 i386
      • rhel5 x86_64
  • Procedure to install ipa server and replicas
    1. on ipaserver: set correct yum repo
    2. on ipaserver: yum install ipa-server
    3. on ipaserver: ipa-server-install , follow instruction
    4. on ipaserver: kinit admin ; ipa-finduser admin , to verify the success of installation
    5. on ipaserver: ipa-replica-prepare ipareplica1.ipaqa.com
    6. on ipaserver: scp replica-info-ipareplica1.ipaqa.com ipareplica1.ipaqa.com:~/.
    7. on ipareplica1: ipa-replica-install replica-info-ipareplica1.ipaqa.com, follow instruction
    8. Done

Use cases

# Use case description result/expection
1 user with valid credential can use ssh login ipa server and any of it's replica server without password prompt
2 valid user can login to any ipa server and it's replica server without setup the local account
3 user can mount NFS once it has valid credential on ipa server and all of its replica server
4 user can change it's password, and ssh login to same server and all of it's replicants with the new password
5 admin can reset user's password, and this user can then change it's password in same ipa server and all of it's replicants
6 when user's password exipred, regardless which server this user access, notification will appear to single user change its password
7 when password policy has been setup, the enforcement of password policy will perform in any ipa server and its replicants whenever user doing password relate operation
8 the following user account information change should always work in any ipa server and it's replicant, and all changes will appear on ipa server and all of it's replicant
Scenario : operation should perform in ipa server or any of its replicant verify in ipa server and all replicant server
1) create a new user verify correctness of data
2) create a new group verify correctness of data
3) delete a user verify correctness of data
4) delete a group verify correctness of data
5) modify a user: add a new filed verify correctness of data
6) modify a user: modify an existing field's value verify correctness of data
7) modify a user: delete an existing field verify correctness of data
8) modify a group: add a new member verify correctness of data
9) modify a group: remove a member verify correctness of data
10) delegation : add a user to group (not clear how to perform it right now) verify user permission 


The test step as below:
   1. ipaserver -> ipareplica verification
          * perform operations below on ipa server
          * verify the same changes appear on replica server
   2. ipareplica -> ipaserver verification
          * perform operations below on ipa replica server
          * verify the same change appear on ipa server
